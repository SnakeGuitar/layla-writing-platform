— 01 —

¿Qué es Layla?

El pitch

Plataforma colaborativa de escritura creativa

Múltiples autores escriben novelas juntos,
gestionan wikis de su universo,
visualizan grafos narrativos
y se comunican por voz en tiempo real.

El problema

Escribir una novela en equipo es un caos

Hoy

Google Docs para el texto.
Discord para hablar.
Notion para personajes.
Hojas de cálculo para la cronología.

Resultado

Cuatro herramientas que no se hablan.
Cambios que se pierden.
Personajes que se contradicen.
Capítulos que no encajan.

La idea

Un solo espacio de trabajo para todo

Manuscrito · Wiki · Grafo · Voz · Roles

en tiempo real, con permisos finos por proyecto.

Capa 3 · Datos

Cada base para lo que sabe hacer mejor

Documentos flexibles.

Neo4j

Mensajería

Los dos backends se hablan por eventos

RabbitMQ · exchange worldbuilding.events

Cuando server-core commitea un nuevo proyecto en SQL,
publica project.created al exchange.

server-worldbuilding consume el evento y inicializa
el documento en MongoDB y los nodos en Neo4j.

Outbox pattern · publicar después del commit, nunca antes

Punto de entrada

Un solo portón al sistema

API Gateway YARP en :5000

/api/users, /api/projects, /hubs/voice → server-core

/api/wiki, /api/graph, /api/manuscripts → server-worldbuilding

Rate limiting · forward de WebSockets · transformaciones de header.
Los clientes nunca saben que hay dos backends.

Modo 1 · ejemplo

Solo para instalar SQL Server...


curl -fsSL https://packages.microsoft.com/keys/microsoft.asc \
  | sudo gpg --dearmor -o /usr/share/keyrings/microsoft.gpg
curl -fsSL https://packages.microsoft.com/config/ubuntu/22.04/mssql-server-2022.list \
  | sudo tee /etc/apt/sources.list.d/mssql.list

sudo apt update && sudo apt install -y mssql-server
sudo /opt/mssql/bin/mssql-conf setup    # interactivo
sudo systemctl enable --now mssql-server

sudo ACCEPT_EULA=Y apt install -y mssql-tools18
sqlcmd -S localhost -U sa -P '***' -C -Q "CREATE DATABASE LaylaCore"

Y eso es solo SQL Server.

Modo 2

Con contenedores

Las VMs siguen existiendo. Pero adentro solo corre Docker.
Cada servicio es una imagen: versionada, sin instalación interactiva.


services:
  sqlserver:
    image: mcr.microsoft.com/mssql/server:2022-latest
    environment:
      - ACCEPT_EULA=Y
      - SA_PASSWORD=${SQL_PASSWORD}
    restart: unless-stopped
  mongodb:
    image: mongo:4.4
  neo4j:
    image: neo4j:5
  rabbitmq:
    image: rabbitmq:3-management

Modo 3

Infraestructura como código

Vagrant

Crea las VMs desde una imagen base reusable.
Define red, recursos, port forwarding.

Puppet

Instala y configura todo dentro de cada VM.
Declarativo: dice qué, no cómo.

Modo 3 · Vagrantfile

Las 3 VMs en un solo archivo


config.vm.box = "bento/ubuntu-22.04"
config.vm.synced_folder "..", "/vagrant"

config.vm.define "data" do |m|
  m.vm.hostname = "layla-data"
  m.vm.network "private_network", ip: "192.168.56.10"
  m.vm.provider("virtualbox") { |vb|
    vb.memory = 3072; vb.cpus = 2
  }
  m.vm.provision "puppet" do |p|
    p.manifest_file = "data.pp"
  end
end
# ... mismo patrón para apps y edge

Modo 3 · Puppet declarativo

Le dices qué, no cómo


package { 'docker-ce':
  ensure  => installed,
}

service { 'docker':
  ensure  => running,
  enable  => true,
  require => Package['docker-ce'],
}

Puppet descubre el "cómo" para Ubuntu, RedHat, lo que sea.

Y para desplegar todo...

vagrant up

3 VMs creadas · Docker instalado · 8 contenedores corriendo

~10 minutos en frío · ~2 minutos con imágenes cacheadas

Bonus

5 bugs que la automatización expuso

#1UseUrls("localhost:...") pisaba ASPNETCORE_URLS
#2UseHttpsRedirection() rompía el proxy YARP
#3Node leía PORT, compose seteaba PORT_HTTP
#4Gateway llamaba UseAuthentication() sin registrarla
#5MongoDB 7+ requiere AVX · VirtualBox no la expone

La automatización no esconde los problemas

Los expone.

Y eso es lo mejor que te puede pasar.
